## Context

当前项目已经有 Compose 原型、部分运行时状态和抓包依据，但页面里仍残留大量静态样板与伪入口。用户希望把首页、订单、设备、我的页改成“看起来像原图”之外，还要“点得到真东西”：任务能跳转、扫码能分流、账号卡片能进独立登录页、设备列表能刷新和编辑。

约束很明确：洗衣链路已有抓包依据，热水链路仍可继续沿用；饮水链路目前缺少完整可执行接口，不能再把洗衣接口硬套过去。

## Goals / Non-Goals

**Goals:**
- 让首页、订单、设备、我的页的主要交互都落到真实状态驱动，而不是静态假按钮。
- 统一扫码后的路由判断，避免饮水码误进洗衣流程。
- 把住理生活 / U净 的登录、绑定设备码、查看状态、更多选项拆成可导航页面。
- 压缩首页信息密度，使主要内容尽量在一屏内可见。
- 保留当前 Compose + runtime 架构，不引入新的重型依赖。

**Non-Goals:**
- 不在本次设计里补齐饮水业务的未验证接口。
- 不重做整体视觉体系，配色与素材沿用当前静态原型方向。
- 不引入 WebView 或原生 XML 页面来绕开 Compose。

## Decisions

1. 任务区采用“单个空状态 + 最多两个活跃任务”的紧凑卡片方案。
   - 推荐方案：当无任务时显示一个居中的“无任务”状态；有任务时按优先级展示 1-2 个任务条，第三个以后折叠为“更多”入口。
   - 备选 A：横向三个小 pill，一屏密度最高，但长文案容易挤压。
   - 备选 B：纵向堆叠任务卡，更清楚但会拉长首页。
   - 选择理由：推荐方案在首页压缩与可读性之间最平衡，也最容易把任务点击跳转做实。

2. 扫码结果采用“类型识别 + 明确兜底”的路由器。
   - 推荐方案：先根据二维码内容中的服务前缀、路径特征或已知字段判定为洗衣/饮水/未知，再进入对应页面。
   - 备选 A：默认进入洗衣，再在失败时回退。这个方案会继续制造误跳转。
   - 备选 B：所有码都进统一预览页，再让用户手动选类型。这个方案更稳，但操作更重。
   - 选择理由：类型识别最贴近用户扫码后的预期，也最符合当前项目已有洗衣/热水/饮水分流的结构。

3. 账户相关操作改成独立页面流，而不是首页内嵌表单。
   - 推荐方案：账号卡片只展示状态摘要，点击后进入独立登录/绑定/状态页，页面内再放手机号、验证码、设备码等输入项。
   - 备选 A：把所有输入都塞回卡片内，页面会继续膨胀。
   - 备选 B：做成底部弹窗，短期更快但不利于查看状态和后续扩展。
   - 选择理由：独立页面最容易压缩首页，也最容易把“查看状态”做成真实反馈。

4. 饮水页面在没有可验证接口前保持“受限功能”状态。
   - 推荐方案：饮水二维码可识别，但进入独立的受限页面或提示页，明确说明当前仅识别到饮水码，后续待接入。
   - 备选 A：伪造一个与洗衣类似的流程，风险最高。
   - 备选 B：完全不处理饮水码，用户会以为扫错了。
   - 选择理由：受限状态能保住链路真实性，也不阻塞后续抓包补齐。

5. 设备列表采用本地编辑 + 下拉刷新双通道。
   - 推荐方案：页面下拉触发刷新，同时显示最近刷新时间；编辑名称、删除设备先作用于本地设备列表，再由运行时尝试同步真实状态。
   - 备选 A：只做静态列表展示，无法满足用户当前需求。
   - 备选 B：强制在线修改，不适合当前验证阶段。
   - 选择理由：本地列表能让 UI 与操作先闭环，后续再接服务端同步。

## Risks / Trade-offs

- [Risk] 饮水功能可能被误认为“已完成” → 在规格中明确写成受限/待接入状态，不伪造洗衣接口。
- [Risk] 任务区内容过多导致首页再次变长 → 限制活跃任务展示数量，其他任务折叠。
- [Risk] 独立页面增多会让导航复杂 → 统一由运行时路由状态管理，减少临时跳转。
- [Risk] 设备列表的本地编辑与真实状态可能不同步 → 在状态文案中区分“本地别名”和“远端设备号”，刷新时重新对齐。
